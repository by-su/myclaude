# Wireframe Architect — PRD-to-Wireframe Generator

PRD(또는 기능 명세)를 분석하여, 실제 프로덕트 팀에서 사용하는 수준의 Lo-fi 와이어프레임을 단일 HTML 파일로 생성한다.

## 핵심 원칙

와이어프레임의 목적은 **시각적 아름다움이 아니라 정보 구조와 사용자 흐름의 검증**이다. 개발자가 이 와이어프레임만 보고도 화면을 구현할 수 있을 만큼 구조가 명확해야 한다. 동시에 디자이너가 이 위에 비주얼을 입힐 수 있도록 레이아웃 결정을 과도하게 하지 않는다.

---

## Phase 1: PRD 분석

PRD를 읽고 다음을 추출한다. PRD에 명시되지 않은 항목은 일반적인 UX 패턴을 기반으로 추론하되, 추론한 부분은 와이어프레임에 `[추론]` 태그로 표시한다.

### 1.1 화면 인벤토리

PRD에서 식별 가능한 모든 화면을 나열한다:
- 화면 이름 (한글 + 영문 ID)
- 화면의 목적 (한 문장)
- 진입 경로 (어디서 이 화면에 도달하는가)
- 핵심 사용자 액션 (이 화면에서 사용자가 하는 주된 행동)

### 1.2 유저 플로우 추출

화면 간의 흐름을 파악한다:
- 메인 플로우 (가장 중요한 사용자 여정)
- 서브 플로우 (설정, 에러 처리, 온보딩 등)
- 각 화면 전환의 트리거 (버튼 클릭, 스와이프, 자동 리다이렉트 등)

### 1.3 컴포넌트 식별

반복적으로 사용되는 UI 패턴을 식별한다:
- 네비게이션 (탭바, 사이드바, 헤더 등)
- 리스트/카드 패턴
- 폼 패턴
- 모달/바텀시트
- 공통 버튼 그룹

## Phase 2: 와이어프레임 설계

### 2.1 설계 원칙

**그레이스케일 전용**: 색상을 사용하지 않는다. `#FFFFFF`, `#F5F5F5`, `#E0E0E0`, `#BDBDBD`, `#9E9E9E`, `#616161`, `#424242`, `#212121` 범위의 그레이만 사용한다. 유일한 예외는 CTA(Call-to-Action) 버튼으로, 이것만 `#333333` 배경 + 흰색 텍스트로 시각적 위계를 표현한다.

**박스 기반 레이아웃**: 모든 UI 요소는 사각형 박스로 표현한다. 둥근 모서리는 `border-radius: 4px`까지만 허용. 이미지 영역은 X 표시가 있는 회색 박스로, 아이콘은 작은 회색 정사각형으로 표현한다.

**실제 콘텐츠 사용**: "Lorem ipsum" 대신 PRD 컨텍스트에 맞는 실제적인 한국어 텍스트를 사용한다. 사용자가 화면을 보고 실제 서비스를 상상할 수 있어야 한다.

**타이포그래피 위계**: 크기와 굵기만으로 정보의 중요도를 표현한다.
- 화면 제목: 24px, bold
- 섹션 제목: 18px, bold  
- 본문: 14px, regular
- 보조 텍스트: 12px, regular, `#9E9E9E`

### 2.2 반응형 전략

각 화면은 모바일(375px)과 데스크탑(1280px) 두 가지 뷰포트로 제공한다. 단순히 크기만 바꾸는 게 아니라, 각 뷰포트에 맞는 레이아웃 패턴을 적용한다:

- **모바일**: 단일 컬럼, 하단 탭 네비게이션, 스택형 카드
- **데스크탑**: 멀티 컬럼 그리드, 사이드 네비게이션, 넓은 테이블/대시보드 레이아웃

중요한 것은 "같은 콘텐츠를 다른 레이아웃으로" 보여주는 것이지, 모바일에서 콘텐츠를 숨기는 것이 아니다.

### 2.3 어노테이션

와이어프레임에는 다음 어노테이션을 포함한다:
- 화면 번호와 이름 (상단 좌측에 고정 라벨)
- 인터랙션 힌트: 클릭 가능한 영역에 점선 테두리 + 화살표 아이콘으로 어디로 이동하는지 표시
- 조건부 표시 요소: 점선 박스 + `[조건: ...]` 라벨
- 추론된 요소: `[추론]` 태그로 PRD에 없지만 추가한 요소 표시

---

## Phase 3: 컴포넌트 명세

와이어프레임 아래에 **컴포넌트 라이브러리** 섹션을 추가한다. 와이어프레임에서 반복 사용되는 모든 컴포넌트를 개별적으로 보여준다.

각 컴포넌트는 다음을 포함:
- **이름**: 예) `ListItem`, `ActionButton`, `SearchBar`
- **시각적 표현**: 실제 크기의 그레이스케일 박스
- **상태(States)**: default, hover, active, disabled, error, loading 등 해당되는 상태들
- **스펙**: 높이, 패딩, 간격 등 주요 수치를 px 단위로 명시
- **사용 화면**: 이 컴포넌트가 사용되는 화면 목록

## Phase 4: HTML 출력

단일 HTML 파일로 모든 와이어프레임을 출력한다.

### 4.1 HTML 구조

```
전체 HTML 파일 구조:
├── 상단 네비게이션 (화면 목록 + 뷰 전환)
├── 개요 섹션
│   ├── 화면 인벤토리 테이블
│   └── 유저 플로우 다이어그램 (텍스트 기반 플로우차트)
├── 화면별 와이어프레임
│   ├── 화면 1 - 모바일 | 데스크탑 (나란히 배치)
│   ├── 화면 2 - 모바일 | 데스크탑
│   └── ...
├── 컴포넌트 라이브러리
│   ├── 컴포넌트 1 (상태별 표현)
│   ├── 컴포넌트 2
│   └── ...
└── 푸터 (생성 정보)
```

### 4.2 인터랙티브 기능

HTML에 다음 인터랙션을 포함한다 (바닐라 JS만 사용, 외부 라이브러리 없음):

- **화면 네비게이션**: 상단 고정 바에서 화면 이름 클릭 시 해당 섹션으로 스크롤
- **뷰포트 전환**: 토글 버튼으로 "모바일만 / 데스크탑만 / 나란히 보기" 전환
- **어노테이션 토글**: 어노테이션(인터랙션 힌트, 조건 라벨 등)을 켜고 끌 수 있는 토글
- **컴포넌트 하이라이트**: 컴포넌트 라이브러리에서 컴포넌트 클릭 시 와이어프레임에서 해당 컴포넌트 사용 위치를 하이라이트

### 4.3 스타일 규칙

```css
/* 와이어프레임 컨테이너 */
전체 배경: #FFFFFF
와이어프레임 배경: #FAFAFA
테두리: 1px solid #E0E0E0

/* 요소 스타일 */
버튼: border 1px solid #BDBDBD, 배경 #F5F5F5
CTA 버튼: 배경 #333333, 텍스트 #FFFFFF
입력 필드: border 1px solid #E0E0E0, 배경 #FFFFFF
이미지 플레이스홀더: 배경 #E0E0E0, 대각선 X 표시
카드: border 1px solid #E0E0E0, 배경 #FFFFFF
구분선: 1px solid #F0F0F0

/* 어노테이션 */
인터랙션 힌트: 점선 border, 색상 #9E9E9E
조건부 요소: 점선 border, [조건] 라벨
추론 표시: [추론] 태그, #BDBDBD 색상

/* 폰트 */
font-family: -apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif
```

### 4.4 유저 플로우 다이어그램

화면 간의 흐름을 텍스트 기반으로 시각화한다. CSS로 박스와 화살표를 그려 플로우차트를 표현한다. 외부 라이브러리는 사용하지 않는다.

```
[시작] → [온보딩] → [홈]
                       ↓
              [상세 페이지] → [결제]
                       ↓
              [마이페이지] → [설정]
```

각 박스는 클릭하면 해당 화면 와이어프레임으로 스크롤된다.

---

## 출력 파일

파일명은 `wireframe-{서비스명(영문 kebab-case)}.html`로 저장한다. 서비스명은 PRD에서 추출한다.

생성 후 사용자에게 다음을 알려준다:
1. 파일 경로
2. 총 화면 수와 컴포넌트 수
3. PRD에서 추론한 부분이 있다면 해당 목록
4. 추가 수정이 필요한 부분에 대한 제안

---

## 실행 체크리스트

1. PRD 파일 읽기 (경로를 사용자에게 확인)
2. Phase 1: PRD 분석 → 화면 인벤토리, 유저 플로우, 컴포넌트 식별
3. 분석 결과를 사용자에게 보여주고 확인 받기 (이 단계에서 누락된 화면이나 잘못된 흐름을 잡을 수 있다)
4. Phase 2-3: 와이어프레임 + 컴포넌트 명세 설계
5. Phase 4: HTML 파일 생성
6. 브라우저에서 열어서 확인할 수 있도록 경로 안내
