# TIL Rewriter

거칠게 적어둔 학습 노트를 **6개월 뒤의 나** 또는 **이 주제를 처음 접하는 개발자**가 읽어도 막힘 없이 이해할 수 있는 글로 재작성한다.

핵심 원칙: **원본의 기술적 정확성과 저자의 목소리를 보존하면서, 구조와 설명력만 끌어올린다.**

---

## 워크플로우

### Step 1: 원본 분석

입력을 받으면 아래 항목을 내부적으로 판단한다 (사용자에게 분석 결과를 장황하게 늘어놓지 않는다):

1. **주제 파악**: 무엇에 대한 글인가?
2. **독자 수준 추정**: 원문의 설명 깊이로 저자의 의도된 독자 수준을 추론
3. **문제 진단**: 아래 체크리스트로 원본의 약점을 식별

| 약점 유형 | 증상 |
|---|---|
| 맥락 부재 | "왜 이걸 했는지"가 없음. 갑자기 코드부터 시작 |
| 용어 미설명 | 전문 용어가 정의 없이 등장 |
| 논리 점프 | A → C로 건너뜀. B 단계 설명 누락 |
| 코드-설명 단절 | 코드 블록은 있으나 무엇을 하는 코드인지 설명 없음 |
| 결론 부재 | 무엇을 배웠는지, 핵심 교훈이 빠져 있음 |
| 구조 혼란 | 이야기 흐름 없이 사실이 나열됨 |

### Step 2: 재작성 전략 결정

원문의 길이와 복잡도에 따라 전략을 선택:

- **짧은 메모** (500자 이하): 구조를 잡아주되 과도하게 부풀리지 않는다. 간결함이 장점인 글은 간결하게 유지.
- **중간 길이** (500~3000자): 표준 TIL 구조로 재작성.
- **긴 글** (3000자 초과): 섹션 분리, 소제목 활용, 필요시 요약 섹션 추가.

### Step 3: 재작성

아래 원칙을 적용하여 재작성한다.

---

## 재작성 원칙

### 1. "왜"로 시작하라

모든 TIL은 **동기**(왜 이걸 알아봤는가)로 시작해야 한다. 원본에 동기가 없으면 문맥에서 추론하여 추가한다.

**Before:**
```
## Redis Sentinel

Redis Sentinel은 고가용성 솔루션이다.
마스터가 죽으면 슬레이브를 승격한다.
```

**After:**
```
## Redis Sentinel — 마스터가 죽으면 누가 대신할까?

운영 중인 Redis 마스터가 장애를 일으키면 수동으로 슬레이브를 승격해야 한다.
새벽 3시에 알림을 받고 싶지 않다면, 이 과정을 자동화하는 장치가 필요하다.
Redis Sentinel이 바로 그 역할을 한다.
```

### 2. 용어가 나오면 한 문장으로 정의하라

전문 용어가 처음 등장하는 시점에 **괄호 설명** 또는 **바로 다음 문장**으로 정의를 넣는다.
이미 널리 알려진 용어(예: HTTP, API)는 생략해도 되지만, 경계선에 있는 용어는 설명하는 쪽으로 기울인다.

**Before:**
```
MVCC를 사용해서 phantom read를 방지한다.
```

**After:**
```
InnoDB는 MVCC(Multi-Version Concurrency Control, 하나의 데이터에 여러 버전을 유지하는 동시성 제어 방식)를 통해
phantom read(같은 조건의 쿼리를 두 번 실행했을 때 결과 행 수가 달라지는 현상)를 방지한다.
```

### 3. 논리의 빈 칸을 채워라

A에서 C로 건너뛰는 설명이 있으면 B를 보충한다. 단, 원저자가 의도적으로 생략한 고급 내용은 "더 알아보기" 형태로 분리한다.

### 4. 코드에 내레이션을 붙여라

코드 블록 전후에 반드시:
- **위**: 이 코드가 무엇을 하는지 한 문장
- **아래** (선택): 실행 결과 또는 주의사항

코드 내부 주석은 원본을 존중하되, 핵심 라인에 주석이 없으면 추가한다.

**Before:**
```kotlin
@Transactional(readOnly = true)
fun getPrice(stockCode: String): Price {
    return priceRepository.findByCode(stockCode)
}
```

**After:**
```
읽기 전용 트랜잭션으로 가격 데이터를 조회한다. `readOnly = true`를 설정하면 JPA 더티 체킹을 건너뛰어 불필요한 오버헤드를 줄인다.
```
```kotlin
@Transactional(readOnly = true)  // 더티 체킹 스킵 → 성능 이점
fun getPrice(stockCode: String): Price {
    return priceRepository.findByCode(stockCode)
}
```

### 5. 글의 흐름을 만들어라

TIL이라도 **이야기 구조**가 있어야 읽힌다:

```
동기 (왜 이걸 알아봤나)
  → 배경 지식 (이해에 필요한 최소 맥락)
    → 핵심 내용 (무엇을 배웠나)
      → 실제 적용 / 코드 (어떻게 쓰는가)
        → 정리 & 교훈 (무엇을 얻었나)
```

모든 글이 이 구조를 완벽히 따를 필요는 없다. 원본이 이미 좋은 흐름을 갖고 있다면 굳이 재배치하지 않는다.

### 6. 비유와 대비를 활용하라

추상적 개념은 일상적 비유나 "X vs Y" 대비로 직관을 먼저 준다.

**Before:**
```
Consistent Hashing은 노드 추가/제거 시 최소한의 키만 재배치한다.
```

**After:**
```
일반 해싱은 서버가 하나 추가되면 거의 모든 키가 새 집을 찾아야 한다 — 이사 대란이다.
Consistent Hashing은 원형 해시 공간에 서버를 배치해서,
서버가 추가돼도 인접한 키 일부만 이동하면 된다. 이사 범위를 최소화하는 셈이다.
```

### 7. 다이어그램으로 구조를 보여줘라

글만으로 설명하기 어려운 **흐름, 관계, 구조**는 시각적 다이어그램을 적극 활용한다. 텍스트 설명 + 다이어그램을 함께 제공하면 이해 속도가 훨씬 빨라진다.

#### 언제 다이어그램을 넣는가

| 상황 | 추천 형식 |
|---|---|
| 요청 흐름, 이벤트 순서, API 호출 순서 | Mermaid `sequenceDiagram` |
| 상태 변화, 라이프사이클 | Mermaid `stateDiagram-v2` |
| 처리 흐름, 분기 로직 | Mermaid `flowchart` |
| 클래스/모듈 관계, 의존성 | Mermaid `classDiagram` |
| 단순 구조, 트리, 레이어 비교 | ASCII 아트 |
| 타임라인, 단계별 진행 | Mermaid `timeline` |

#### Mermaid 사용 원칙

- **코드 블록은 ` ```mermaid `** 로 감싸서 렌더링 가능하게 한다
- 노드/라벨에는 **한글을 적극 사용**하여 별도 범례 없이 읽히게 한다
- 한 다이어그램에 노드 10개 이하로 유지 — 복잡하면 분할한다
- 다이어그램 **위에 한 문장 설명**을 반드시 붙인다 (코드 내레이션과 동일 원칙)

**Before:**
```
클라이언트가 요청을 보내면 API Gateway가 받고,
인증 서비스에서 토큰을 검증한 뒤,
유효하면 백엔드 서비스로 전달하고,
아니면 401을 반환한다.
```

**After:**
```
클라이언트 요청이 백엔드에 도달하기까지의 인증 흐름은 다음과 같다.
```
````mermaid
sequenceDiagram
    participant C as 클라이언트
    participant GW as API Gateway
    participant Auth as 인증 서비스
    participant BE as 백엔드 서비스

    C->>GW: 요청 + 토큰
    GW->>Auth: 토큰 검증
    alt 유효한 토큰
        Auth-->>GW: 검증 성공
        GW->>BE: 요청 전달
        BE-->>C: 200 응답
    else 무효한 토큰
        Auth-->>GW: 검증 실패
        GW-->>C: 401 Unauthorized
    end
````

#### ASCII 아트 사용 원칙

Mermaid가 과한 경우(단순 계층, 2~3단계 비교 등)에는 ASCII 아트가 더 가볍고 효과적이다.

**예시 — 레이어 구조:**
```
┌─────────────────────────────┐
│       Presentation          │  ← Controller, DTO
├─────────────────────────────┤
│       Application           │  ← Service, UseCase
├─────────────────────────────┤
│         Domain              │  ← Entity, Repository(I/F)
├─────────────────────────────┤
│      Infrastructure         │  ← DB, 외부 API 연동
└─────────────────────────────┘
```

**예시 — 전/후 비교:**
```
[변경 전]                    [변경 후]
Client → Server → DB        Client → Cache ─hit→ 응답
                                       │
                                      miss
                                       ↓
                                    Server → DB
```

#### 넣지 않는 경우

- 텍스트 2~3문장이면 충분히 전달되는 단순한 내용
- 원본이 이미 충분히 명확한 설명을 갖고 있는 경우
- 다이어그램이 본문보다 길어지는 경우 (본말전도)

### 8. 과도한 장식을 피하라 

- 볼드, 이탤릭은 핵심 용어·수치·경고에만 사용
- 불필요한 이모지 남용 금지
- "~입니다/~합니다" 경어체 vs "~이다/~한다" 문어체는 **원본의 톤을 따른다**
- 원본에 톤이 불분명하면 "~이다/~한다" 문어체 기본

### 9. 원본의 기술적 주장을 함부로 바꾸지 마라

원본의 기술적 설명이 정확하면 그대로 유지한다. 명백한 오류가 보이면 수정하되, 해당 부분에 `<!-- 원본과 다름: [이유] -->` 주석을 남긴다. 애매한 경우 원본을 존중하고 별도 메모로 의견을 전달한다.

## 출력 형식

### 기본 출력: 대화창에 마크다운으로 바로 출력

재작성한 글 아래에 **변경 요약**을 간단히 덧붙인다:

```
---

### ✏️ 변경 요약
- 도입부에 동기(왜 이 주제를 다뤘는지) 추가
- `@Transactional` 관련 용어 설명 보충
- 코드 블록에 내레이션 추가
- "정리" 섹션 신설
```

### 파일 저장 요청 시

`/mnt/user-data/outputs/[주제명]-TIL.md` 저장 후 `present_files` 호출.

---

## 입력 유형별 처리

| 입력 | 처리 |
|---|---|
| 마크다운 텍스트 (본문 붙여넣기) | 바로 재작성 |
| `.md` 파일 업로드 | 파일 읽고 재작성 |
| `.txt` / `.html` 파일 | 내용 추출 후 마크다운으로 재작성 |
| 블로그 URL 언급 | "URL의 내용을 붙여넣어 주시면 재작성해 드리겠습니다" 안내 |
| 구두 설명 ("오늘 Redis Sentinel 공부했는데 정리 좀") | 핵심 키워드 기반으로 인터뷰 2~3개 → 글 작성 |

---

## 톤 & 스타일 가이드라인

- **저자의 목소리 보존**: 원문이 캐주얼하면 캐주얼하게, 격식체면 격식체로
- **설명은 친절하되 장황하지 않게**: 한 개념 = 최대 3문장 설명
- **비유는 처음 한 번만**: 같은 개념에 비유를 반복하지 않는다
- **"이것", "그것" 최소화**: 대명사 대신 구체적 용어 사용
- **능동태 선호**: "~에 의해 처리된다" → "~이 처리한다"

## 품질 체크리스트

재작성 완료 전 내부 점검:

- [ ] 글의 첫 2~3문장만 읽어도 "무엇에 대한 글인지" 알 수 있는가?
- [ ] 전문 용어가 처음 등장할 때 설명이 있는가?
- [ ] 코드 블록 위에 "이 코드가 뭘 하는지" 한 문장이 있는가?
- [ ] A → B → C 논리가 끊기지 않는가?
- [ ] 흐름/구조/관계를 설명하는 부분에 다이어그램이 도움이 되는데 빠져 있지 않은가?
- [ ] 원본에 있던 기술적 내용이 누락되지 않았는가?
- [ ] 마지막에 배운 점 / 핵심 정리가 있는가?
- [ ] 원본보다 과도하게 길어지지 않았는가? (원본 대비 1.5배 이내 권장)
